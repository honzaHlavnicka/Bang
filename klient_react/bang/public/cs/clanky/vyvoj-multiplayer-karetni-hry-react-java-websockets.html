Jak jsem naprogramoval online karetní engine (a nezbláznil se u toho)

Napsal Honza | Vývojář & Autor

Vytvořit online tahovou karetní hru zní na papíře docela jednoduše. Hráč A zahraje kartu, hráč B ztratí život. Pohoda, že? Jenže ve chvíli, kdy se do toho pustíte, zjistíte, že musíte řešit odpojování hráčů, asynchronní dialogy, tahy napříč složitým polem (topologie Bangu!) a synchronizaci stavu mezi serverem a klienty. V tomto článku vás provedu technologiemi, které jsem pro svůj engine zvolil.

Architektura: Striktní Klient-Server

Už od prvního řádku kódu jsem věděl, že klient (webový prohlížeč) nesmí nic rozhodovat. Nikdy nevěřte klientovi. Pokud bych nechal prohlížeč počítat životy nebo míchat karty, okamžitě by to svádělo k podvádění. Můj prohlížeč slouží čistě jako "hloupý" prezentační terminál. Pošle na server požadavek „Hráč Honza chce zahrát kartu Bang! na hráče Adama“ a čeká, co mu server milostivě dovolí vykreslit.

Backend: Proč zrovna Java 11 a WebSockets?

Serverovou část jsem napsal v Javě 11. Důvod byl pragmatický – potřeboval jsem, aby to běželo na našem školním serveru (Gymnázium Arabská), který novější verzi tehdy nepodporoval. A upřímně, silně typovaný jazyk je pro složitou herní logiku záchrana.

Pro komunikaci jsem zavrhl běžné HTTP dotazy (polling). Neustále se ptát serveru „Už hraju? Už hraju?“ je neefektivní. Místo toho jsem použil knihovnu Java-WebSocket. Díky WebSocketům mám trvalé obousměrné spojení. Server může okamžitě říct klientovi „Hele, přišel jsi o život, začni krvácet!“ bez jakékoliv latence.

Můj komunikační protokol je naprosto primitivní, ale funkční textový formát. Vypadá to nějak takto: <typZpravy>:<data>. Když se podíváte do konzole prohlížeče při hraní, uvidíte přesně, co se děje. Formát se sice postupem času trochu zvrhl, ale pro debugování ručním psaním zpráv je to k nezaplacení.

Vzkříšení mrtvých spojení (Reconnect logic)

Největší oříšek, který jsem musel rozlousknout, byly výpadky spojení. Když hrajete Bang! už hodinu a najednou vám spadne Wi-Fi, máte chuť prohodit počítač oknem. Můj KomunikatorHry na serveru to naštěstí řeší. Každému hráči při připojení vygeneruje unikátní token, který si klient uloží do localStorage. Když hráč vypadne, server jeho postavu nezabije, ale podrží ji "v limbu" až 5 minut. Jakmile se hráč vrátí s platným tokenem, server mu obratem pošle masivní JSON se vším, co propásl, a hra plynule pokračuje dál.

Frontend: záchrana jménem React a TypeScript

Úplně první verze klienta byla ve Vanilla JavaScriptu. Byla to obrovská špageta kódu. Velmi rychle jsem přešel na React, Vite a TypeScript.

TypeScript mi zachránil spoustu nervů při parsování dat ze serveru. Reactí Context API (v mém případě GameContext) funguje jako dokonalé centrální úložiště herního stavu. A co se týče vizuálu? Pro plynulé přetahování karet myší nebo prstem po displeji jsem sáhl po knihovně react-dnd. Nastavit Drag & Drop tak, aby fungoval logicky (kartu z ruky jde hodit na odhazovací balíček, nebo vyložit k sobě, nebo s ní zacílit hráče), byla docela výzva, ale výsledek za to stojí.
