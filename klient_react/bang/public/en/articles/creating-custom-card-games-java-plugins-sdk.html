How I Refactored My Engine for Plugins (And How You Can Add Your Own Game)

Written by Honza | SDK Author

In the very beginning, my engine was completely hardcoded for Bang!. But as I added more and more logic, I realized a fundamental truth: I am way too lazy to write a new WebSocket handler, turn queue manager, and frontend UI for every single new game (like UNO). It was time to completely decouple the games from the engine.

I decided to build a system of dynamic plugins. Today, my core server has no idea how Bang! or UNO is actually played. It serves merely as a massive sandbox into which you drop your rules in the form of a packaged JAR file.

ClassLoader Magic

So, how does it work? In the server directory, there's a folder called /plugins/. When the server boots up, the PluginLoader class scans this directory. Using Java's built-in URLClassLoader, it peers into all the JAR files looking for compiled classes that implement my HerniPlugin (GamePlugin) interface.

When it finds a match, it instantiates it. Thanks to this, deploying a new game to production is trivial—just toss the new JAR into the folder, and the server picks it up. No complex core refactoring needed.

Your First Game: Meet the SDK

If you want to code your own game, I've prepared a full SDK (Software Development Kit), complete with JavaDocs. The core of your game will be a class that implements HerniPravidla (GameRules). This is where the engine "asks" what it should do.

Here are a few key methods you'll override as a game developer:

  • pripravBalicek() (Prepare Deck): This is where you instantiate your cards and shove them into the engine's main draw pile.
  • getViditelnePrvky() (Get Visible UI Elements): Not every game needs a health bar. Here, you tell the client which UI panels to activate (e.g., discard pile, player's board).
  • hracChceUkoncitTah() (Player Wants to End Turn): This is where your logic runs. If you're playing UNO and the player hasn't drawn or played a card, this method returns false, and the engine refuses to end their turn.

Object-Oriented Cards: Playable vs. Deployable

Every card in the system is a full object with its own ID. If you're programming a card that a player throws straight from their hand to the pile (like a +2 in UNO), you implement the HratelnaKarta (PlayableCard) interface. The main odehrat() (Play) method triggers the moment a user drags and drops the card in their browser. Inside this method, you evaluate if the card can be played, and immediately apply its effects.

If it's an item that stays on the board (like a weapon or a barrel), you implement VylozitelnaKarta (DeployableCard). The engine automatically handles moving the card to the correct visual slot on the client's screen.

The hardest part of Bang! was handling the Mustang and Scope cards, because they dynamically alter circle distances. I solved this by giving my Player class the ability to calculate modulo distances, while card effects simply apply modifiers (+1 or -1) to those calculations.
