How I Built an Online Card Game Engine (Without Losing My Mind)

Written by Honza | Developer & Creator

Creating a turn-based online card game sounds pretty simple on paper. Player A plays a card, Player B loses a life. Easy, right? But the moment you start coding, you realize you have to deal with player disconnects, asynchronous dialogs, turn queues, topological distance calculations (looking at you, Bang!), and state synchronization. In this article, I’ll walk you through the tech stack I chose for my engine.

Architecture: Strict Client-Server

From line one, I knew the client (the web browser) couldn't be trusted with any decisions. Never trust the client. If I let the browser calculate health or shuffle decks, it would immediately invite cheating. My frontend serves purely as a "dumb" presentation terminal. It sends a request to the server saying, "Player Honza wants to play a Bang! card on Player Adam," and waits to see what the server graciously allows it to render.

The Backend: Why Java 11 and WebSockets?

I wrote the server in Java 11. The reason was highly pragmatic—I needed it to run on my high school's server, which didn't support newer versions at the time. Honestly, a strongly typed language is an absolute lifesaver for complex game logic.

For communication, I scrapped standard HTTP polling. Constantly asking the server "Is it my turn yet?" is terribly inefficient. Instead, I used the Java-WebSocket library. Thanks to WebSockets, I have a persistent, bidirectional connection. The server can instantly tell the client, "Hey, you just lost a life, start bleeding!" with near-zero latency.

My communication protocol is a primitive but effective text format. It looks something like this: <messageType>:<data>. If you look at the browser console while playing, you'll see exactly what's happening. The format got a bit chaotic over time, but for debugging by manually typing payloads, it’s priceless.

Resurrecting Dead Connections (Reconnect Logic)

The hardest nut to crack was handling dropped connections. When you've been playing Bang! for an hour and your Wi-Fi suddenly dies, you want to throw your PC out the window. Fortunately, my server's KomunikatorHry (Game Communicator) handles this. When a player connects, they are assigned a unique token stored in localStorage. If they drop, the server doesn't kill their character—it holds them "in limbo" for up to 5 minutes. Once the player returns with a valid token, the server immediately shoots them a massive JSON with everything they missed, and the game continues seamlessly.

The Frontend: Falling in Love with React and TypeScript

The very first version of the client was written in Vanilla JavaScript. It was a massive plate of spaghetti code. I quickly migrated to React, Vite, and TypeScript.

TypeScript saved my sanity when parsing server data. React’s Context API (my GameContext) acts as a perfect central repository for the game state. And for the visuals? For smooth card dragging with a mouse or finger, I used react-dnd. Configuring Drag & Drop to feel logical (throwing a card to the discard pile, equipping it to your board, or targeting another player) was quite the challenge, but the result is incredibly satisfying.
